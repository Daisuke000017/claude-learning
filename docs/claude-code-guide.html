Module 1 - Lesson 1

コーディングアシスタントとは？

コーディングアシスタントとは、言語モデルを使ってコードを書いたり開発タスクを実行するツールです。Claude Codeはその中でも、ツール使用能力に優れたAIアシスタントです。

タスク受領

バグ修正やエラーメッセージなど、開発タスクを受け取ります

コンテキスト収集

ファイルを読み、コードベースの構造を理解します

計画策定

問題解決のための実行計画を立てます

アクション実行

ファイル更新、テスト実行などを行います

重要な前提知識

言語モデルの制限

言語モデル自体はテキストの入出力のみ可能です。ファイルの読み書きやコマンド実行は直接できません。そこで「ツール使用」という仕組みが必要になります。

Claudeモデルの優位性

Claudeは他の言語モデルと比較してツール使用能力が高いのが特徴です。ツールの機能理解や複合的なタスク実行に優れ、拡張性も高い設計です。

Module 1 - Lesson 1

ツール使用の仕組み

言語モデルがファイル操作やコマンド実行を行うための核心的な仕組みが「ツール使用システム」です。全てのコーディングアシスタントがこの仕組みを基盤としています。

リクエスト

ユーザーの依頼にアシスタントが指示を追加

ツール呼び出し

モデルが必要なアクションをフォーマットで応答

実行

アシスタントが実際のアクションを実行

結果返送

実行結果をモデルに送り返す

最終応答

モデルが結果を解釈して応答を生成

ツール使用の流れ // 1. ユーザー：「main.jsのバグを修正して」 // 2. Claude：read_file("main.js") を呼び出し // 3. アシスタント：ファイルの内容を読み取って返す // 4. Claude：修正内容を決定し write_file() を呼び出し // 5. アシスタント：ファイルを更新 → 完了を報告

ファイル読み書き

プロジェクト内のファイルを直接読み取り、編集・作成できます。

コマンド実行

シェルコマンドを実行し、ビルド・テスト・デプロイなどを行います。

コード検索

外部サーバーに送信せず、ローカルで直接コード検索。セキュリティも安全です。

Module 2 - Lesson 1

コンテキスト管理

コンテキスト（文脈情報）の管理はClaude Codeの効果を左右する最も重要な要素です。多すぎる情報はパフォーマンスを低下させ、少なすぎると正確な作業ができません。

1

/init コマンド

コードベース全体を分析し、プロジェクト構造・アーキテクチャ・重要ファイルをまとめたClaude.mdファイルを自動生成します。

2

# メモリーモード

「#」キーを押すとClaude.mdファイルを自然言語で編集できるモードに入ります。チームの規約やよくある指示を記録しましょう。

3

@ ファイル参照

「@ファイル名」で特定ファイルを直接参照。Claudeに検索させるより正確で効率的にコンテキストを提供できます。

Claude.mdの3つのレベル

Machine レベル

全プロジェクト共通のグローバル設定。個人の作業スタイルを定義

Project レベル（共有）

チームで共有。ソースコントロールにコミットして全員が同じ指示を利用

Local レベル（個人）

個人用の設定。コミットされず自分だけの指示やカスタマイズを保持

ベストプラクティス

データベーススキーマなど頻繁に参照するファイルは、Claude.mdに記載しておくと毎回のリクエストに自動的に含まれます。「必要な情報を適切な量だけ」がClaude Codeの鉄則です。

Module 2 - Lesson 2

コード変更の管理

Claude Codeを使った効率的なコード変更ワークフローと、品質を高めるための2つの特別モードを解説します。

スクリーンショット連携

Control + V（macOSではCommand+Vではなく）でスクリーンショットを貼り付けると、Claudeが画面の内容を理解し、UI修正や改善に活用します。問題箇所を視覚的に伝える最速の方法です。

パフォーマンスブーストモード

Plan モード

広さを担当 - より多くのファイルを調査し、詳細な実装計画を作成してから実行します。複数ファイルにまたがるタスクに最適。

Shift + Tab を2回

Thinking モード

深さを担当 - 拡張推論の予算を使い、複雑なロジックやデバッグに集中的に取り組みます。トリッキーなバグ修正に最適。

「Ultra think」と入力

使い分けガイド

Plan = 多ファイル横断タスク、リファクタリング、新機能追加
Think = 難解なバグ、アルゴリズム設計、複雑なロジック
併用 = 大規模かつ複雑なタスク（追加トークンコストに注意）

Git連携：Claude Codeは変更のステージング・コミットも実行でき、内容を理解した説明的なコミットメッセージを自動生成します。

Module 2 - Lesson 3

コンテキスト制御テクニック

長時間の作業やタスク切り替え時に、会話の品質を維持するための5つのテクニックを紹介します。

Esc

Escape - 中断

Claudeの応答中に1回押すと即座に中断。出力の方向性を修正したいときに使います。

E+#

Escape + メモリー

中断後に#キーでメモリーに記録。同じミスの繰り返しを防止する強力な組み合わせです。

2x

ダブルEscape - 巻き戻し

Escapeを2回押すと過去のメッセージ一覧が表示され、任意のポイントに戻れます。無関係なデバッグのやり取りを会話からスキップできます。

Compact コマンド

会話全体を要約しつつ、Claudeが学んだ知識を保持。会話が長くなりすぎた時のクリーンアップに最適です。

Clear コマンド

会話履歴を完全削除してゼロからスタート。まったく異なるタスクに切り替える時に使います。

MCPサーバーによる機能拡張

MCP（Model Context Protocol）サーバーは、Claude Codeに外部ツールとの連携能力を追加する仕組みです。ローカルまたはリモートで動作し、開発ワークフローを大幅に拡張します。

インストール

権限承認

初回使用時に権限を承認。settings.local.json で自動承認も設定可能

活用

追加されたツールを使って複雑な自動化ワークフローを実行

実践例

Playwright MCPでUI自動改善

Playwright MCPサーバーを使い、Claudeがlocalhostのアプリをブラウザで開いてUIコンポーネントを生成。スタイリング品質を視覚的に分析し、生成プロンプトを自動修正。繰り返すことで大幅にスタイリングが改善されました。

視覚フィードバックによる自動改善

Module 3 - Lesson 1

Hooksの基本概念

Hooksとは、Claudeがツールを実行する前後に自動的に走るカスタムコマンドです。ツールの動作を監視・制御し、自動フィードバックループを構築できます。

Pre-hook
実行前チェック
ツール実行
read / edit / grep
Post-hook
実行後処理

Pre-tool use Hook

ツール実行前に起動。操作内容を検査し、条件によってはブロック（阻止）できます。エラーメッセージをClaudeに返すことも可能。

Post-tool use Hook

ツール実行後に起動。フォローアップ処理を実行し、結果をフィードバックとしてClaudeに返します。ブロック機能はありません。

活用例

自動フォーマット

ファイル作成後にPrettier等を自動実行

テスト実行

編集後にユニットテストを自動実行

ファイル保護

機密ファイルへのアクセスをブロック

型チェック

TypeScript編集後に型エラーを自動検出

Module 3 - Lesson 2-4

Hooksの定義と実装

Hookの設定ファイル構造と、実際の実装方法を詳しく解説します。.envファイルの読み取りをブロックする実例で学びましょう。

設定ファイル構造

settings.local.json { "hooks": { "preToolUse": [ { "matcher": "read|grep", "command": "node ./hooks/read_hook.js" } ], "postToolUse": [ { "matcher": "edit", "command": "node ./hooks/format_hook.js" } ] } }

終了コードの意味

終了コード意味動作
exit 0許可ツール呼び出しを通常通り実行
exit 2ブロックPre-hookのみ。ツール実行を阻止し、stderrのメッセージをClaudeに返す

実装例：.envファイルの保護

hooks/read_hook.js // stdinからツール呼び出しデータをJSON形式で受信 const chunks = []; process.stdin.on('data', c => chunks.push(c)); process.stdin.on('end', () => { const data = JSON.parse(Buffer.concat(chunks)); // ファイルパスを取得 const filePath = data.input?.file_path || data.input?.path || ''; // .envファイルへのアクセスをブロック if (filePath.includes('.env')) { console.error('Blocked: .env file access is not allowed'); process.exit(2); // ブロック } process.exit(0); // 許可 });

注意点

Hookの設定を変更した後は、必ずClaude Codeを再起動してください。ファイルパスのチェック時は、readツールのfile_pathとgrepツールのpathの両方に対応するフォールバック処理が必要です。

Hookが受け取るデータ

JSON入力 { "session_id": "abc-123", "tool_name": "read", "input": { "file_path": "/project/.env" } }

Module 3 - Lesson 5-6

実用的なHooksパターン

Claude Codeの弱点を自動的に補う2つの強力なHooksパターンを紹介します。これらのパターンで型エラーとコード重複という頻出問題を解決できます。

Post-hook

TypeScript型チェッカーHook

問題：Claudeが関数シグネチャを変更した際、呼び出し側のコードを更新し忘れて型エラーが発生する。

解決策：TypeScriptファイル編集後に tsc --no-emit を自動実行。型エラーを検出するとフィードバックとしてClaudeに返し、自動修正させます。

型付き言語なら何でも適用可能
Post-hook

コード重複防止Hook

問題：複雑なタスクでClaudeが既存のクエリ/関数を再利用せず、新たに重複コードを作成してしまう。

解決策：監視対象ディレクトリ（例：queries/）の変更を検出し、別のClaudeインスタンスをSDK経由で起動。既存コードとの重複を分析し、発見した場合はexit 2でブロック。

重要ディレクトリに限定推奨

共通パターン：自動フィードバックループ

両方のHookに共通するのは「Post-hookで検査 → エラー発見 → Claudeにフィードバック → 自動修正」というパターンです。

注意：追加のAPI呼び出しが発生するため、コスト・時間のトレードオフを考慮してください。

Module 3 - Lesson 7

Claude Code SDK

Claude Code SDKは、Claude Codeの機能をプログラムから利用するためのインターフェースです。CLI・TypeScript・Pythonの3つのライブラリが用意されており、既存のパイプラインに知能を組み込めます。

CLI

コマンドラインから直接実行。シェルスクリプトやCI/CDパイプラインに最適

TypeScript

Node.jsアプリに統合。型安全なAPIでフルコントロール

Python

データ分析パイプラインや自動化スクリプトに組み込み

権限モデル

デフォルト（読み取り専用）

  • ファイルの読み取り
  • ディレクトリの参照
  • Grep（コード検索）
  • 書き込み・編集は不可
VS

書き込み許可設定後

  • ファイルの編集・作成
  • コマンド実行
  • 全ツールの使用が可能
  • options.allowTools で指定
TypeScript SDK 例 import { query } from '@anthropic/claude-code'; const result = await query({ prompt: "このファイルのバグを修正して", options: { allowTools: ["edit", "write"] // 書き込み権限を追加 } });

最適な用途

SDKはヘルパーコマンド、自動化スクリプト、Hooks内でのサブプロセスなど、既存プロジェクト内のパイプラインに組み込んで使うのが最も効果的です。スタンドアロンでの使用より、既存ワークフローへの統合に向いています。
